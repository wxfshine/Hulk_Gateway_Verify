# Hulk_Gateway_Verify 使用指南

本文档介绍如何在本地环境运行 Hulk_Gateway_Verify 脚本、配置常用项、以及如何查看测试结果与排查常见问题。

## 项目简介

Hulk_Gateway_Verify.py 是一个用于校验本地或内网 LLM Gateway（兼容 OpenAI 风格 API）的自动化验证脚本。
脚本包含对常用接口的批量调用能力：对话（chat/completions）、文本续写（completions）、向量嵌入（embeddings）和反馈链路（feedback）。
运行后会把每条调用结果写入 CSV，同时生成基于 CSV 的 HTML 分析报告，便于统计和排查。

## 环境与依赖

- Python：推荐 3.8 及以上
- 依赖：requests

安装依赖：

```powershell
pip install requests
```

建议使用虚拟环境（venv、pipenv、poetry）隔离依赖。

## 配置（避免把敏感信息写入源码）

脚本顶部存在可修改常量：

- base_url：网关地址，例如 `http://hulk.cmit.local:18080/v1`
- api_key：Bearer Token（请不要把真实 Key 明文提交到仓库）
- EMBEDDING_MODEL_NAMES / CHAT_MODEL_NAMES：用于筛选要测试的模型

建议通过环境变量配置敏感信息，并在脚本中读取，例如：

Windows PowerShell：

```powershell
$env:HULK_API_KEY = "<your_api_key>"
$env:HULK_BASE_URL = "http://hulk.cmit.local:18080/v1"
python Hulk_Gateway_Verify.py
```

Linux / macOS：

```bash
export HULK_API_KEY="<your_api_key>"
export HULK_BASE_URL="http://hulk.cmit.local:18080/v1"
python Hulk_Gateway_Verify.py
```

如果短期调试需要，也可暂时在脚本中修改 base_url 与 api_key，但请勿将含有真实凭据的文件提交到版本控制。

## 运行方式

在仓库根目录运行：

```bash
python Hulk_Gateway_Verify.py
```

脚本流程概览：
1. 初始化日志（输出同时写入 log 目录下的时间戳日志文件）
2. 调用 GET /models 获取模型列表并按白名单筛选
3. 针对不同 compliance_check 值执行 chat/completion/embeddings/feedback 测试
4. 将每条测试记录写入 CSV（log/hulk_gateway_verify_result_<timestamp>.csv）
5. 测试结束后生成 HTML 分析报告（log/hulk_gateway_verify_analysis_<timestamp>.html），并尝试在默认浏览器打开

## 产物说明

- 日志目录：`log/`（包含运行日志、CSV、HTML 报告）
- 结果 CSV：`log/hulk_gateway_verify_result_<timestamp>.csv`，字段包括 timestamp、status、model_name、compliance_check、input_content、api_name、result_or_error
- 分析报告：`log/hulk_gateway_verify_analysis_<timestamp>.html`，包含整体统计、按接口/模型汇总、错误分布、不匹配记录等

## 常用调整点

- 测试文本：修改脚本中 test_chat_completion/test_completion/test_embedding 的 contents 列表
- 模型白名单：修改脚本顶部的 CHAT_MODEL_NAMES / EMBEDDING_MODEL_NAMES
- 输出目录：默认在脚本同级的 log 子目录，可按需修改

## 常见问题与排查建议

- 无法获取模型列表：检查 base_url、api_key 是否正确，网关是否可达；必要时在浏览器或 curl 中直接访问 GET <base_url>/models 以确认返回
- 网络超时或请求失败：当前脚本未统一设置 requests 超时时间与重试策略，网络不稳定时会抛出异常。建议局部修改 post_json/get_json 增加 timeout 和重试（requests.Session + urllib3 Retry）
- CSV 包含大量原始输入文本：若输入过长，会影响阅读与统计，可在 record_test_result 中存储输入摘要（如前 200 字）或将完整文本保存到独立文件并在 CSV 中引用
- 反馈校验误报：test_feedback 采用 submit -> query 的全等匹配策略，后端可能会增加 id/timestamp 或调整字段，导致查不到刚提交的记录。更稳健的方式是：用 submit 返回的 id 查询，或做模糊匹配（model、rating、comment、时间窗口）
- 日志或 HTML 无法打开：确认脚本有写权限、log 目录存在以及是否在无 GUI 的环境中运行（默认浏览器无法打开）

## 建议改进（可选）

- 从环境变量读取 api_key 并在缺失时代替硬编码报错或提示
- 使用 requests.Session，统一设置 timeout 和重试策略以提升鲁棒性
- 强化 extract_* 函数的输入校验与容错，避免对非预期响应结构抛出未捕获异常
- 增加命令行参数（argparse），支持：仅运行某类测试、指定并发度、设置请求超时
- 为关键函数（HTTP 层、解析层）添加单元测试（requests-mock / responses）以覆盖异常路径

## 扩展用法（合并历史结果 / 生成统一报告）

仓库提供了将多个 CSV 合并生成统一分析报告的思路（脚本中有合并逻辑占位），常见操作：

- 合并多个 CSV：在本地脚本中实现合并后再调用 analyze_test_results
- 生成统一 HTML 报告：把合并后的 CSV 文件路径传给 analyze_test_results 并调用 generate_analysis_html

（当前命令行参数支持视脚本变更而定；若需要我可帮助将这些功能做成可用命令行参数）
